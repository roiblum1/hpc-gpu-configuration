# PARAMETERS.md — every knob node-foundation turns, and why

A deep dive on **each parameter this chart changes on the GPU nodes**: what it does, what
the default was, why we chose this value, what breaks if it's wrong, and how to check it.
Written to be readable without prior context. Companions: [CLAUDE.md](CLAUDE.md) (editing
guidance), [BIOS.md](BIOS.md) (firmware half, out of band), [HYPERSHIFT.md](HYPERSHIFT.md)
(delivery on Hosted Control Planes), [README.md](README.md) (chart usage).

Every parameter here can be checked on a live node with
`SSH_KEY=... USER_SSH=... PREFIX=<prefix> scripts/verify-nodes.sh`.

## 1. How the pieces fit

One chart, four delivery mechanisms, all scoped to the `gpu-hpc` pool:

| Mechanism | Carries | Why this vehicle |
|---|---|---|
| **PerformanceProfile** | cpusets, NUMA policy, hugepages, kernel args, IRQ policy | The one API that makes NTO generate a *consistent* set: kernel args + KubeletConfig + parent tuned profile + the `performance-gpu-hpc` RuntimeClass. Hand-assembling those four drifts. |
| **Tuned child profile** | sysctls + THP mode | The PerformanceProfile API has no sysctl field; a child profile (`include=` the generated one) is the supported extension point. |
| **MachineConfig (CRI-O memlock)** | one CRI-O config drop-in | Container runtime config isn't kernel tuning; it's a file on disk → MachineConfig. |
| **MachineConfig (RoCE QoS)** | a systemd oneshot re-pinning NIC QoS every boot | NIC QoS state doesn't survive reboots/driver reloads; a boot-time service is the only reliable owner. |

Everything below reboots the pool node-by-node when changed. **Batch your edits.**

## 2. Pool scope

### `pool.name` / `pool.roleLabel` = `gpu-hpc`

- **What:** the MachineConfigPool that receives all of this, and the node role label
  (`node-role.kubernetes.io/gpu-hpc`) that selects nodes into it.
- **Why:** kernel/tuning rollouts must not touch the rest of the fleet, and every other
  chart (GPU operator, rails, scheduler, serving) selects nodes by this label — it is the
  §10 node-selection invariant. One pool per hardware generation keeps per-SKU tuning possible.
- **If wrong:** config lands on non-GPU nodes (or nowhere), and upper layers schedule onto
  untuned nodes — failures surface phases later as "Dynamo is slow".
- **Check:** `oc get mcp gpu-hpc` shows your GPU nodes; `oc get node -l node-role.kubernetes.io/gpu-hpc`.

## 3. PerformanceProfile

The kubelet settings people expect to write by hand (`cpuManagerPolicy`, `topologyManagerPolicy`,
`memoryManagerPolicy`, `reservedSystemCPUs`) are **generated by NTO from the fields below** —
there is no `spec.kubeletConfig`, and adding your own KubeletConfig would conflict (§8).

### `cpu.reserved` = `0-7,56-63`

- **What:** CPUs kept for "housekeeping": kubelet, CRI-O, systemd, and — critically — NIC IRQs.
  Becomes `reservedSystemCPUs` in the generated kubelet config and `systemd.cpu_affinity` on boot.
- **Why this value:** **8 cores per socket, on BOTH sockets, deliberately.** Each rail NIC hangs
  off one socket; its interrupts must be served by cores on the *same* socket or every packet
  completion crosses the UPI inter-socket link. Reserving one socket's worth (e.g. `0-15`) looks
  tidier but silently taxes socket-1's four rails. The values assume 2×56 cores with SMT off
  (112 logical CPUs) — for other shapes run `scripts/verify-nodes.sh generate`, which derives
  the first 8 cores of each socket from the node's real topology.
- **SMT caveat:** if SMT is on, reserved/isolated must contain complete sibling pairs, or
  kubelet's `full-pcpus-only` rejects the GPU pods at admission (generate mode handles this).
- **If wrong:** rail IRQs land on isolated cores (jitter on the serving hot loop) or cross
  sockets (latency), or GPU pods are rejected outright.
- **Check:** verify-nodes.sh checks 4 and 7 (`reservedSystemCPUs`, PID 1 affinity);
  `/proc/interrupts` shows rail-NIC IRQs on local-socket reserved cores.

### `cpu.isolated` = `8-55,64-111`

- **What:** everything else — the exclusive pool kubelet's static CPU manager hands to
  Guaranteed pods with integer CPU requests. NTO generates `isolcpus=managed_irq,…`,
  `nohz_full=…`, `rcu_nocbs=…` from it.
- **Why this value:** the complement of reserved. 96 CPUs = the integer `cpu` request of one
  full-node GPU worker pod (8 GPUs + 8 rail VFs + all isolated CPUs = one pod per node).
- **If wrong:** overlap with reserved is rejected by NTO; a gap wastes cores silently.
- **Check:** verify-nodes.sh check 2 (`isolcpus`/`nohz_full` in `/proc/cmdline`).

### `numa.topologyPolicy` = `best-effort`

- **What:** becomes kubelet's `topologyManagerPolicy` — how strictly kubelet insists that a
  pod's CPUs, memory, and devices come from one NUMA node.
- **Why this value and NOT `single-numa-node`:** our worker pods are **full-node by design**
  (8 GPUs on two sockets). A pod that spans both sockets *cannot* satisfy `single-numa-node`
  (or `restricted`) — kubelet would reject it at admission, always. `best-effort` admits the
  pod; *intra-pod* NUMA correctness is delegated to the runtime, which pins per-GPU/per-NIC
  locally (rail map: GPU n ↔ NIC n ↔ its socket). Do not "harden" this value — it is the
  §1.2 decision, and stricter is strictly worse here.
- **If wrong:** `single-numa-node` → GPU pods stuck `Pending`/`TopologyAffinityError` forever.
- **Check:** verify-nodes.sh check 4 (`topologyManagerPolicy: best-effort`).

### `hugepages` = 16 × 1G (default size 1G)

- **What:** boot-time-reserved 1 GiB pages (`default_hugepagesz=1G hugepagesz=1G hugepages=16`
  generated by NTO), usable by pods via `hugepages-1Gi` resources.
- **Why this value — small on purpose:** the KV-cache host tier (KVBM) uses **CUDA pinned
  (page-locked) memory, not hugetlbfs**. Reserving "the size of the DRAM tier" in hugepages
  (a common reflex) *steals* the RAM the tier needs — hugetlbfs pages are unusable for normal
  allocations. 16 GiB is a cushion for the few components that genuinely map hugetlbfs.
  Grow only if something measurably consumes it.
- **If wrong:** too big → the CPU cache tier ordering (`GPU-KV ≤ CPU ≤ disk`, §10) breaks
  because pod memory can't fit the CPU tier; too small → hugetlbfs consumers fail to map.
- **Check:** verify-nodes.sh check 3 (`nr_hugepages` = 16, per-NUMA split as INFO).

### `globallyDisableIrqLoadBalancing` = `false`

- **What:** `true` would banish *all* device IRQs to reserved cores globally, node-wide.
  `false` keeps normal IRQ balancing, and lets **individual pods** opt their exclusive CPUs
  out of IRQ delivery.
- **Why `false`:** this is one half of the **per-pod low-latency contract** (§3.3). The other
  half lives on the GPU worker pods in the serving chart: `runtimeClassName:
  performance-gpu-hpc` **plus** the `irq-load-balancing.crio.io: "disable"` and
  `cpu-quota.crio.io: "disable"` annotations. Pod-scoped opt-out keeps every other pod on the
  node schedulable and served normally. **Remove either half and IRQ isolation silently
  degrades** — no error, just jitter.
- **If wrong:** `true` starves reserved cores with all IRQ load and breaks the contract's
  assumptions; `false` *without* the pod-side annotations means no isolation at all.
- **Check:** the RuntimeClass exists (`oc get runtimeclass performance-gpu-hpc`); worker pods
  carry both annotations; verify-nodes.sh check 6 confirms the CRI-O runtime drop-in.

### `net.userLevelNetworking` = `false`

- **What:** `true` would size NIC queues down to the reserved-CPU count (for DPDK-style
  userspace networking).
- **Why `false`:** RDMA is kernel-bypass but not DPDK — the rails need their full queue
  spread. Leave the NIC queue layout alone.

### `workloadHints` = `{realTime: true, highPowerConsumption: false, perPodPowerManagement: false}`

- **What:** selectors for which tuning bundles NTO's generated profile activates. Pinned
  explicitly in `values.yaml` even though these *are* the NTO defaults — because §4.2's
  pruned kernel-arg list depends on `realTime: true` being in effect.
- **`realTime: true`** → NTO generates `nohz_full`, `nosoftlockup`, `skew_tick=1`,
  `rcutree.kthread_prio=11`, and (via the CPU-vendor include) `tsc=reliable nmi_watchdog=0
  mce=off`; plus RT-leaning sysctls and the `stalld` starvation daemon. This is where most
  of the "classic low-latency args" actually come from.
- **`highPowerConsumption: false`** → NTO does *not* add `idle=poll` or its own C-state cap;
  we cap C-states with our two kernel args instead (reversible per fleet, §4.1).
- **`perPodPowerManagement: false`** → tuned pins `governor=performance`,
  `energy_perf_bias=performance`, `min_perf_pct=100` and `force_latency=cstate.id:1|3` —
  the *runtime* half of the C-state cap (see §4.1).
- **If wrong:** flipping `realTime` to `false` silently drops seven latency args from the
  next render — nothing errors, the node just gets slower and jittery.
- **Check:** verify-nodes.sh check 2 (the generated args appear on `/proc/cmdline`).

## 4. Kernel boot arguments

### Why "PerformanceProfile parameters" show up inside a Tuned object (the machinery)

NTO never writes kernel args directly. It **compiles the PerformanceProfile into a
generated Tuned profile** — `openshift-node-performance-gpu-hpc`, visible with
`oc get tuned -n openshift-cluster-node-tuning-operator` — whose `[bootloader]` section
carries the cmdline, and whose `[sysctl]`/`[cpu]`/`[service]` sections carry the runtime
tuning. TuneD's bootloader plugin hands the args to MCO, which reboots the pool. So when
you inspect the cluster and find cpusets and kernel args "under a Tuned", that is the
**compiled artifact of the PerformanceProfile, not duplication** — and it is exactly why
our own `gpu-hpc-extras` child profile (§5) does `include=openshift-node-performance-gpu-hpc`:
it layers on top of the compiled output. The generated profile itself includes
`openshift-node,cpu-partitioning` plus a CPU-vendor include (`…-intel-x86` / `…-amd-x86`).

The full expected `/proc/cmdline` is therefore **our five args (§4.1) + NTO's generated
args (§4.2)**. verify-nodes.sh check 2 asserts both groups.

### 4.1 Args this chart sets (`additionalKernelArgs` — only what NTO does not generate)

#### `numa_balancing=disable`

- **Default:** auto-NUMA balancing on (RHEL default).
- **Why:** auto-NUMA samples memory access and *migrates* pages toward the CPU touching
  them. Our placement is already static (pinned CPUs, per-GPU/per-NIC NUMA-local buffers);
  the sampler only adds page faults and migration jitter, and it fights CUDA pinned memory.
- **If missing:** periodic latency spikes under memory pressure; `numa_pages_migrated`
  climbing in `/proc/vmstat`.

#### `pcie_aspm=off`

- **Default:** firmware/OS may enable PCIe Active State Power Management.
- **Why:** ASPM lets PCIe links doze; the *exit* latency (µs → ms) lands directly on
  GPUDirect RDMA and NVMe completions as spikes. GPU nodes never idle enough to profit.
- **If missing:** intermittent GDR/NVMe completion spikes that profile as "random".

#### `rcu_nocb_poll`

- **Default:** offloaded-RCU kthreads are woken by IPIs from the CPUs they serve.
- **Why:** pairs with the NTO-generated `rcu_nocbs=<isolated>` (§4.2): the `rcuo` kthreads
  *poll* for callbacks instead of being IPI-woken by isolated cores — removes the last
  regular IPI source on the hot cores. Poll cost lands on reserved cores, where it belongs.
- **If missing:** low-rate IPI noise on isolated cores; visible as rare ITL outliers.

#### `intel_idle.max_cstate=1` + `processor.max_cstate=1`

- **Default:** with `highPowerConsumption: false`, NTO does **not** cap C-states at boot —
  deep C-states stay allowed as far as the kernel is concerned.
- **Why:** caps idle at C1 so cores never pay deep-sleep wakeup latency (tens of µs)
  mid-decode-step. This is the **boot-time layer of a three-layer cap**: BIOS C-state policy
  ([BIOS.md](BIOS.md)) → these args (survive tuned crashes/restarts) → tuned's
  `force_latency=cstate.id:1|3` + `governor=performance` (runtime, from
  `perPodPowerManagement: false`). On EPYC, `intel_idle.max_cstate` is inert and
  `processor.max_cstate` carries the cap.
- **Trade-off:** higher idle power. If idle thermals outweigh tail latency, drop both args
  and use the per-pod `cpu-c-states.crio.io` annotation instead — that's a supported,
  documented alternative, which is why these two stay in *our* list rather than a hint.

### 4.2 Args NTO generates — expected on the node, **never** re-add them by hand

Two owners of one arg = silent drift when the cpusets change (the §1.2 rule). This chart's
earlier revisions listed six of these in `additionalKernelArgs`; they were removed after
auditing the 4.20 NTO templates.

**From the cpusets (always generated):**

- **`isolcpus=managed_irq,<isolated>`** — kernel keeps *managed* device IRQs off the isolated
  CPUs. Note: `managed_irq` only, not `domain` — scheduler placement stays kubelet's job
  (static CPU manager), not the kernel's.
- **`nohz=on` + `nohz_full=<isolated>`** *(nohz_full via the realTime hint)* — tickless
  isolated cores: no 250 Hz scheduler tick on a core running exactly one busy thread.
- **`rcu_nocbs=<isolated>`** — RCU callbacks are offloaded from isolated cores to `rcuo`
  kthreads (which our `rcu_nocb_poll` then makes poll).
- **`tuned.non_isolcpus=<mask>`** — bookkeeping mask telling tuned which CPUs are
  housekeeping; drives its IRQ-affinity handling.
- **`systemd.cpu_affinity=<reserved>`** — PID 1 and every service it spawns start pinned to
  reserved cores. This is what verify-nodes.sh check 7 asserts.
- **`default_hugepagesz=1G hugepagesz=1G hugepages=16`** — boot-time reservation from §3's
  hugepages block.

**From `realTime: true` (base template):**

- **`nosoftlockup`** — no soft-lockup detector activity on isolated cores. A 100%-busy
  polling worker looks *exactly* like a soft lockup to the detector; without this you get
  false-positive stack dumps that themselves cause jitter.
- **`skew_tick=1`** — de-synchronizes per-CPU ticks; with 100+ cores, simultaneous ticks
  are a periodic all-core latency pulse.
- **`rcutree.kthread_prio=11`** — runs RCU kthreads at RT priority 11 so grace periods
  can't be starved by busy workloads.

**From the CPU-vendor include (`realTime: true`, Intel/AMD):**

- **`tsc=reliable`** — declares the TSC trustworthy, disabling the clocksource watchdog
  that periodically cross-checks it. Two effects: no watchdog overhead, and no risk of a
  false-positive mid-run demotion to HPET (catastrophically slow timekeeping). Safe on any
  modern Xeon/EPYC with invariant TSC (`constant_tsc nonstop_tsc` in `/proc/cpuinfo`).
  **Is it necessary?** Not strictly — recent kernels auto-qualify well-behaved Intel TSCs —
  but NTO ships it with the realTime hint, so *it's not ours to decide*: it arrives
  generated. We used to list it explicitly; removed as a duplicate.
- **`nmi_watchdog=0`** — hard-lockup (NMI) watchdog off: no periodic NMIs on any core.
  (This is why our old `nowatchdog` arg was redundant: `nmi_watchdog=0` + `nosoftlockup`
  together cover both detectors.)
- **`mce=off`** — disables machine-check *polling* (the periodic corrected-error scans),
  another periodic all-core interference source. Uncorrected fatal MCEs still trap.
- **`intel_iommu=on iommu=pt`** (Intel) / **`iommu=pt`** (AMD) — IOMMU on for SR-IOV VFs,
  but *passthrough* for host-driver DMA: GPUDirect RDMA and NVMe skip per-mapping IOTLB
  translation, which measurably taxes GDR bandwidth. Generated by the vendor include —
  another pair we used to set by hand.
- **`intel_pstate=<recommended>`** (Intel; generation-dependent `active`/`disable`) /
  **`amd_pstate=guided`** (AMD) — P-state driver mode; paired with tuned's
  `governor=performance`. verify-nodes.sh reports the actual value as INFO.

- **Check (both groups):** verify-nodes.sh check 2 asserts every expected string in
  `/proc/cmdline`; the cpuset-bearing args are compared against the expected cpusets.

## 5. Tuned child profile — sysctls

Delivered by the `gpu-hpc-extras` Tuned profile, which `include=`s the NTO-generated profile
and layers what the PerformanceProfile API can't express. Priority 19 (lower number wins)
so it takes over *while inheriting everything*.

Inherited unchanged from the generated profile (listed so you know where they come from):
`net.ipv4.tcp_fastopen=3`, `vm.dirty_ratio=10`, `vm.dirty_background_ratio=3`, and — from
the realTime hint — `kernel.hung_task_timeout_secs=600`, `kernel.nmi_watchdog=0`,
`kernel.sched_rt_runtime_us=-1`, `vm.stat_interval=10`. Our child profile only **adds or
overrides** the entries below.

### `vm.swappiness=0` (RHEL default 60; the generated profile already lowers it to 10 — we override to 0)

Never swap under memory pressure if there is any file cache left to reclaim. With ~1 TB of
CUDA-pinned memory, swapping anything the inference stack touches means multi-ms stalls.
`10` still permits swap as a last resort; `0` says "only when the alternative is OOM".
(Pinned pages themselves can't swap — this protects everything around them.)

### `vm.zone_reclaim_mode=0` (default 0 — pinned explicitly)

Off = a NUMA node that runs out of free pages borrows from the other node instead of
stalling to reclaim locally. Local reclaim stalls are exactly the jitter we spent §3/§4
removing; pinned here so no profile/driver ever flips it back on.

### `vm.max_map_count=1048576` (default 65 530)

Max memory mappings per process. Loading a 754B-parameter model means thousands of mmap'd
weight shards + CUDA allocations; the default is sized for ordinary apps and overflows with
cryptic `ENOMEM`-on-mmap failures mid-load.

### `net.core.rmem_max` / `wmem_max` = 536870912 (default ~208 KiB)

Ceiling (512 MiB) for socket buffers. Serves the **TCP paths only** — model staging pulls,
control plane, metrics; RDMA bypasses TCP entirely. Big ceilings let bulk staging transfers
on 100/400G links actually fill the pipe.

### `net.ipv4.tcp_rmem` / `tcp_wmem` = `4096 87380/65536 268435456`

TCP autotuning min/default/max. Default per-connection behavior is unchanged (middle value
stays standard); only the autotune *max* is raised to 256 MiB so long-fat staging transfers
can grow. Raising the middle would balloon memory on thousands of idle connections.

### `fs.aio-max-nr=1048576` (default 65 536)

Max in-flight async I/O requests system-wide. The NVMe KV-cache disk tier and model staging
both drive deep async queues; overflow surfaces as `io_setup: EAGAIN` — an error message
three layers away from its cause.

### `vm.min_free_kbytes=4194304` (default: tens of MiB, sqrt-scaled)

A 4 GiB free-page reserve. With ~1 TB pinned (unreclaimable) plus 9000-MTU ring buffers,
atomic allocations (NIC rings, drivers — contexts that cannot wait for reclaim) start failing
under pressure with the default reserve. 2–4 GiB is the standard figure for RDMA-heavy hosts
this size. Don't set it dramatically higher: the reserve is permanently unusable RAM.

### ARP-flux suppression: `arp_ignore=1`, `arp_announce=2`, `arp_filter=1` (defaults 0/0/0)

- **What:** Linux's default "weak host model" answers ARP requests for **any** local IP on
  **any** interface, and may source ARP with whichever address it likes. On a multi-homed
  host (8 rail NICs + management), NICs end up advertising each other's addresses — "ARP flux".
  - `arp_ignore=1` — only answer if the target IP is configured **on the interface the
    request arrived on**.
  - `arp_announce=2` — always source ARP from the best local address **for that interface**,
    never a sibling NIC's address.
  - `arp_filter=1` — reply only if the kernel would actually **route** the answer out of the
    interface the request came in on (this is what DGX OS ships by default).
- **Why:** deterministic L2: each rail answers for exactly its own address on its own wire.
  On the reference rail plan (one /24 per rail over /31 point-to-point links —
  [sriov-rails/ROUTING.md](../sriov-rails/ROUTING.md)) cross-rail flux mostly can't happen at
  L3, but this is standard multi-homed RDMA hardening and it costs nothing. Set on `conf.all`,
  which the kernel combines (max/OR) with per-interface values — covers current *and* future
  host interfaces.
- **Scope caveat (important):** sysctls are **per network namespace**. This covers the *host*
  netns. The rail VFs live inside **pod** network namespaces — if your rail NADs put several
  rails in one subnet, set the same trio pod-side via the `tuning` CNI meta-plugin in the NAD
  (rails are user-provided; this chart can't do it for you).
- **If wrong:** intermittent wrong-MAC learning upstream, traffic for rail X arriving on rail
  Y, hard-to-reproduce RDMA connection resets.
- **Check:** verify-nodes.sh check 5.

### `transparent_hugepages=madvise` (RHEL default `always`; the generated profile sets `never` — we override to `madvise`)

`always` lets khugepaged rewrite any process's memory into 2M pages in the background —
compaction stalls at unpredictable times, which is why NTO's generated profile goes all the
way to `never`. But `never` throws away real wins for allocators that *ask* for THP
explicitly (`madvise(MADV_HUGEPAGE)` — CUDA/PyTorch allocators do). `madvise` = huge pages
only where the application opted in: khugepaged works only on opted-in regions. A
deliberate, documented override of the generated value — middle ground, on purpose.

## 6. CRI-O memlock

### `default_ulimits = ["memlock=-1:-1"]` (drop-in via MachineConfig)

- **What:** removes the locked-memory rlimit for every container CRI-O starts.
- **Why:** RDMA (ibverbs) **registers** memory — pins it and hands the NIC its physical
  addresses. Registration counts against `RLIMIT_MEMLOCK`, and the runtime default (tens of
  MiB) is nothing next to the gigabytes NCCL/UCX register. The failure is famously indirect:
  `ibv_reg_mr` fails deep inside NCCL/UCX init and surfaces three layers up as
  "NCCL error: unhandled system error" on pod start.
- **Why not per-pod?** Pod `securityContext` cannot raise rlimits; the runtime default is the
  supported lever. It applies to all containers on these GPU-dedicated nodes — acceptable
  blast radius, and the §10 "memlock unlimited" row.
- **Check:** verify-nodes.sh check 6; from inside any pod on the node: `ulimit -l` → `unlimited`.

## 7. RoCE QoS (RoCE fabrics only)

Delivered as a systemd oneshot ([files/roce-qos.sh](files/roce-qos.sh)) that re-pins every
value on **every rail PF, identically, every boot** — NIC QoS state does not survive reboots
or driver reloads, so a boot-time service is the only reliable owner. Values are §10
invariants: **the switch fabric and the pod NCCL/UCX env must mirror them exactly.**

> **InfiniBand environments (`env/h100-*-ib` branches): this whole section is off**
> (`roceQos.enabled: false`). IB is lossless by credit-based flow control; QoS lives in the
> fabric (SLs / subnet manager). There is no host DSCP/PFC/ECN/CNP to pin.

### `--trust dscp`

Classify traffic by the L3 DSCP field, not L2 PCP (VLAN priority bits). DSCP survives
routing hops and works without VLAN tags — PCP does neither. Prerequisite for everything
below; also why DCBX must stay **off** in NIC firmware ([BIOS.md](BIOS.md)) — otherwise the
switch can renegotiate trust out from under this script.

### PFC vector `0,0,0,1,0,0,0,0` (`pfcPriority: 3`)

Priority-based Flow Control on exactly one priority: **3**, the lossless class RoCE rides.
One lossless class is deliberate — every additional PFC-enabled priority multiplies
head-of-line-blocking and buffer-tuning complexity on the switch. The switch must enable PFC
on the same priority with matching buffer thresholds.

### `traffic_class = 106` → DSCP 26 + ECN (`trafficClass: 106`)

The number that must be identical in **four places** (§10 triple): here (host),
`NCCL_IB_TC=106` (collectives), `UCX_IB_TRAFFIC_CLASS=106` (KV-cache transfers), and the
switch's classification. The arithmetic: traffic-class byte = DSCP<<2 | ECN bits →
26×4 + 2 (ECT(0)) = **106**. DSCP 26 maps to priority 3 (the PFC class); the ECT bits mark
packets ECN-capable so the switch can signal congestion by marking instead of dropping.
**If NCCL or UCX disagrees, that engine silently rides the lossy best-effort queue** — no
error, just terrible tail latency under load.

### `cma_roce_tos -t 106`

Same value, third enforcement point: connections set up through **RDMA-CM** (how UCX/NIXL
typically establishes QPs) take their ToS from this per-device setting, not from
`tc/1/traffic_class`. Without it, KV-cache transfers land on the wrong queue even though
NCCL is correct — the half-configured state that's hardest to notice.

### CNP: `cnp_dscp = 48`, `cnp_802p_prio = 6`

Congestion Notification Packets — the "slow down" messages of ECN — are marked DSCP 48
(CS6, network-control) and sent at priority 6. CNPs must **outrun** the data they're trying
to slow: they ride a *separate, high-priority, non-lossless* queue that the switch schedules
strict-priority. A CNP stuck behind the congestion it reports is useless. Switch CNP queue
config must mirror these two numbers.

### DCQCN enables: `roce_np/enable/3 = 1`, `roce_rp/enable/3 = 1`

The host half of ECN-based congestion control, on the lossless priority: **NP** (notification
point — receiver turns ECN-marked arrivals into CNPs) and **RP** (reaction point — sender
rate-limits when CNPs arrive). This pair *is* "enabling ECN" host-side; the switch does the
actual WRED/ECN marking (out of band, must be configured to mark, not drop, on the RoCE
queue). PFC then exists only as the *backstop* for the microbursts DCQCN can't catch —
DCQCN does the steady-state work.

### `ethtool -A rx off tx off`

Global (802.3x) link pause **off**. Global pause and per-priority PFC are mutually exclusive
in practice: global pause freezes the whole port — including the CNP queue — defeating both
the priority separation and the congestion control above.

### `ip link set mtu 9000`

§10 invariant: MTU 9000 **end-to-end** — PF (here), `SriovNetworkNodePolicy.mtu`, the pod
NADs, and every switch port. RoCE fragments nothing: a single hop with MTU 1500 doesn't
degrade — it breaks, with maddeningly intermittent symptoms (small messages pass, large
collectives stall).

- **Check (whole section):** `mlnx_qos -i <rail>` shows trust=dscp + PFC prio 3;
  `cat /sys/class/net/<rail>/ecn/roce_np/cnp_dscp` = 48 on every rail;
  `systemctl status roce-qos.service` after boot.

## 8. What we deliberately do NOT set

Just as load-bearing as what we do set. Each of these is a "someone will suggest it in
review" item — the answer is already decided:

- **No hand-written KubeletConfig, no `spec.kubeletConfig`.** The PerformanceProfile API has
  no such field (the API server would prune it), and NTO *generates* the KubeletConfig
  (cpuManager static + full-pcpus-only, memoryManager Static, topologyManager from `numa.
  topologyPolicy`, reservedSystemCPUs from `cpu.reserved`). A second KubeletConfig on the
  same pool = two owners fighting. Escape hatch for fields the profile can't express:
  the `kubeletconfig.experimental` annotation on the PerformanceProfile.
- **No hand-written `isolcpus` / `nohz_full` / `rcu_nocbs` / hugepage args.** NTO generates
  them from the cpusets. Writing them again in `additionalKernelArgs` creates duplicate args
  that drift apart silently on the next cpuset change.
- **Not `single-numa-node`.** Full-node pods span both sockets by definition; stricter
  topology policy = permanent admission failure (§3).
- **Hugepages not sized to the DRAM tier.** KVBM uses CUDA pinned memory, not hugetlbfs (§3).
- **Not `globallyDisableIrqLoadBalancing: true`.** Per-pod opt-out via the RuntimeClass +
  CRI-O annotations is the contract; global banishment punishes every pod on the node (§3).
- **No realtime kernel (`realTimeKernel.enabled` stays off).** Inference wants low *jitter*,
  not hard-RT guarantees; the RT kernel costs throughput and complicates the NVIDIA driver
  stack. The C-state caps + IRQ isolation above capture the win without the cost.
- **`rp_filter` untouched.** Strict reverse-path filtering would break asymmetric multi-rail
  routing; OpenShift's default (loose where it matters) is left alone. The ARP trio (§5)
  handles the L2 hygiene without touching L3 path validation.
- **No `mlxconfig`/BIOS settings in this chart.** Firmware is out of band by design →
  [BIOS.md](BIOS.md). A chart that half-owns firmware invites drift between what's declared
  and what's flashed.
