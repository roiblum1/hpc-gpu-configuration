# Validating and Optimizing a Rail-Optimized RoCEv2 Design for NCCL/UCX on Air-Gapped OpenShift 4.20 + Dell XE9680 HGX

## TL;DR
- **Your design is largely correct and will run fast, but the flat shared `192.168.0.0/24` remote route is the one architectural wart: it is only "safe" because each pod's netns sees a single rail's route. Re-segment to a distinct remote prefix per rail index (rail *n* → its own subnet via rail *n*'s `/31` next-hop). This is the canonical fix and the only change that materially reduces cross-rail leakage/ECMP risk.**
- **Your QoS encoding is actually correct and is the industry-canonical RoCEv2 value: traffic-class 106 = DSCP 26 (AF31) left-shifted by 2 with ECN bits = ECT(0). DSCP 26 << 2 = 104; the +2 sets ECN-capable transport so DCQCN works. NVIDIA's HPC-X UCX guidance explicitly sets `UCX_IB_TRAFFIC_CLASS=106` for RoCEv2. GID index 3 = RoCEv2/IPv4 on mlx5 — correct, but verify inside the pod.**
- **The biggest performance levers are not your IP plan: enforce RDMA `exclusive` subsystem mode for SR-IOV, confirm GPUDirect RDMA peer-to-peer works on the same-PCIe-switch GPU↔NIC pair (ACS must be off on that path even though SR-IOV wants IOMMU on), keep PCIe relaxed ordering on, and run ECN/DCQCN as the primary congestion control with PFC tuned as a per-hop safety net.**

## Key Findings

### 1. The flat `/24` shared remote route — functionally OK per-pod, but not the correct design
The authoritative word comes directly from Sylvain Jeaugey (NCCL maintainer) in NVIDIA/nccl issue #1109: *"The rail-optimization is about physical switch wiring. Not about the IP subnet design."* He notes a distinct subnet per rail "might work for most use cases… but there can be corner cases where it will cause failures because we'll need to communicate across NICs," and that a single flat subnet for all NICs "requires complex configuration in linux to make sure RoCE will select the right NIC." The original report in that thread is the canonical broken case: with all NICs in one broadcast domain, *"NCCL picks only one 400G ethernet NIC and Switch perspective MAC and ARP reply is seen on one interface per node"* — i.e., single-NIC ARP collapse.

Your situation is different from that broken case because you do **not** put all 8 rails in one namespace. Each pod gets exactly one NAD = one rail = one route in its network namespace. So in the per-pod case, the flat `192.168.0.0/24 via <rail-n leaf /31 peer>` works: the pod has only one egress interface and one route, so there is no ambiguity, no ECMP, and the BGP-routed leaf does the actual forwarding to the remote GPU-NIC. This is why your `ib_send_bw` tests pass.

**However, the flat /24 is the wrong canonical model and you should re-segment**, for these reasons:
- **L3 rail isolation / NCCL_CROSS_NIC=0 enforcement.** A distinct remote prefix per rail makes "rail *n* can only reach rail *n* endpoints" a property of the routing table, not just of pod scheduling. It turns your logical rail discipline into an enforced L3 invariant.
- **Any multi-rail-in-one-namespace scenario breaks the flat plan.** If you ever (a) run a host-network diagnostic pod, (b) attach multiple rail NADs to one pod (e.g., for PXN-style multi-NIC, NIXL/UCX multi-rail, or `NCCL_IB_HCA=mlx5_0,mlx5_1`), or (c) test from the host netns, that namespace now holds 8 routes to the *same* `192.168.0.0/24` via 8 different next-hops. Linux collapses these into ECMP and hashes flows across rails — directly violating `NCCL_CROSS_NIC=0`, causing cross-rail traffic, and defeating the rail-optimized topology.
- **No downside given your constraints.** You explicitly said manageability and CR count don't matter and the fabric is fully routed with BGP. Re-segmenting costs only configuration, which you've already said you'll happily do.

**Canonical per-rail subnet plan:** assign each rail index its own remote destination prefix, reachable only via that rail's `/31` next-hop. For example rail0 remote endpoints in `192.168.10.0/24`, rail1 in `192.168.11.0/24`, … rail7 in `192.168.17.0/24` (the exact prefixes don't matter; the one-prefix-per-rail mapping does). Each rail's NAD then carries exactly one route: `dst <rail-n remote prefix> via <rail-n leaf /31 peer>`. Mirror this on the switch side so BGP advertises each rail's remote endpoints only within that rail's address block.

### 2. `/31` point-to-point with the leaf port as gateway — correct for a routed RoCE fabric
This is the right model. A fully routed (L3-to-the-host) RoCE fabric with eBGP is exactly what NVIDIA Spectrum-X, Cisco's "AI/ML routed fabric," and the Broadcom/Dell designs use. The Cisco *Nexus 9000 Series Switches for AI Clusters* white paper describes the backend as an AI/ML routed fabric type that *"uses Exterior Border Gateway Protocol (eBGP) for providing IP reachability between eight backend NIC ports on all the GPU nodes of a cluster"* (Cisco measured ~390 GB/s AllReduce busbw across 64 GPUs over this eBGP fabric). `/31` per RFC 3021 is the standard for numbered point-to-point links and is exactly what Broadcom documents for its MI300X RoCE networking guide ("Single Leaf Switch Topology with 31-bit Subnets"). Using the leaf switch port as the `/31` peer/gateway is correct. L3-to-the-host wins over L2/PFC-flooded designs because it gives clean ECMP, fast BGP convergence, and avoids L2 broadcast-domain ARP-collapse issues (the very problem that triggered the NCCL #1109 thread, where all NICs in one broadcast domain caused single-NIC ARP collapse).

### 3. RoCEv2 QoS: your encoding is correct (and is the industry default)
The traffic-class/DSCP math is the single most-misunderstood point, so here it is precisely. The 8-bit IP Type-of-Service / IPv6 Traffic-Class byte is `[DSCP:6 bits][ECN:2 bits]`. Therefore **traffic_class = DSCP << 2** (or DSCP × 4), with the low 2 bits carrying ECN.

- DSCP 26 (binary `011010`, this is AF31) << 2 = `01101000` = **104** = 0x68 (pure DSCP, ECN = 00).
- Your value **106** = `01101010` = DSCP `011010` (=26) + ECN `10` (=ECT(0)). So **106 = DSCP 26 with the ECN-capable-transport bit set**, not an error.

This is intentional and canonical: NVIDIA's `cma_roce_tos` guidance says to compute DSCP×4 and "add 2 to include the ECN," and NVIDIA's HPC-X UCX documentation explicitly instructs configuring the fabric as lossless and *setting `UCX_IB_TRAFFIC_CLASS=106`*, noting *"By default, RoCE v2 and IPv4 are used, if available."* DSCP 26 for RoCE data + DSCP 48 for CNP is the de-facto standard across NVIDIA, Broadcom (`roce_dscp` default 26, `cnp_dscp` 48), and AMD/Arista example configs (`qos map traffic-class 3 to dscp 26`). **Verdict: traffic-class 106 is exactly right. Keep it.** Just make sure NCCL is being given the *full TOS byte* (106) via `NCCL_IB_TC`, not the bare DSCP (26) — `NCCL_IB_TC` takes the traffic-class byte, so 106 is correct; setting it to 26 would mean DSCP 6 and land in the wrong queue.

**GID index 3 = RoCEv2/IPv4 — correct on mlx5.** On a ConnectX/mlx5 port with one IPv4 address, the GID table is: index 0 = RoCEv1 link-local, 1 = RoCEv2 link-local, 2 = RoCEv1 IPv4, **3 = RoCEv2 IPv4** (IPv4-mapped IPv6 `::ffff:a.b.c.d`). Multiple `show_gids` outputs confirm index 3 is the RoCEv2 IPv4 entry. **Caveat:** the GID index is *not* guaranteed stable — it depends on NIC firmware, RoCE versions enabled, and the number of IPs, and Broadcom explicitly warns indices can change on link flap or firmware change. Inside an SR-IOV pod with `exclusive` RDMA mode, the VF's GID table is small and index 3 should still be RoCEv2 IPv4, but you must verify with `show_gids`/`ibv_devinfo -v` *inside the pod*. Better still, NVIDIA's NCCL networking troubleshooting guidance states that *"with NCCL 2.21 and later the GID index is dynamically selected… this environment variable should not be set"* — so leaving `NCCL_IB_GID_INDEX=-1` (auto), with NCCL selecting the RoCEv2 GID via `NCCL_IB_ROCE_VERSION_NUM=2` (default), is more robust than hard-pinning 3.

**PFC vs ECN/DCQCN on a routed fabric.** On a BGP-routed L3 fabric, ECN/DCQCN must be your primary congestion control and PFC a tuned per-hop safety net. The consensus (Juniper, Cisco, WWT, NVIDIA) is DCQCN = ECN (marks early, end-to-end, survives L3 because the ECN bits live in the IP header) + PFC (per-hop, link-local pause, triggered *after* ECN). PFC does work across routed hops — it is sent hop-by-hop from congestion point toward the source and uses DSCP-based priority — but PFC alone risks head-of-line blocking and pause-propagation storms, and ECN alone reacts too slowly. Tune ECN/WRED thresholds *below* the PFC threshold so ECN engages first. The critical requirement: PFC, ECN thresholds, and DSCP-to-queue maps must be **identical end-to-end** (NIC firmware + every leaf). CNP (DSCP 48) must be classified to a strict-priority, never-dropped queue.

### 4. SR-IOV + RDMA on OpenShift — the highest-leverage correctness items
- **RDMA subsystem MUST be in `exclusive` mode.** For SR-IOV (not shared/macvlan), set `rdmaMode: exclusive` in a `SriovNetworkPoolConfig`. This gives each pod its own isolated RDMA device and GID table in its network namespace — essential for correct per-rail GID selection and isolation. Shared mode (`rdmaSharedDevicePlugin`) is for macvlan/ipvlan and is the wrong choice for your dedicated-NIC-per-pod model. This corresponds to the host `rdma system set netns exclusive` / `options ib_core netns_mode=0` setting.
- Your `SriovNetworkNodePolicy` settings are correct: `deviceType: netdevice`, `isRdma: true`, `numVfs: 1` (whole-GPU, no MIG), MTU 9000.
- **MTU 9000 must be truly end-to-end** — VF, pod, and every leaf port (switches typically need MTU ~9216 to carry a 9000 payload plus headers; e.g., Arista/Cisco examples use 9214/9216). A single mismatched hop silently kills PFC pause-frame generation and tanks RoCE throughput. Note that mlx5 will not pass jumbo frames unless the underlying device MTU is set correctly.
- **NCCL per-rail environment (whole-node, rail-pinned):**
  - `NCCL_CROSS_NIC=0` — correct for rail-optimized (keeps each ring/tree on the same rail).
  - `NCCL_IB_HCA=mlx5_<n>:1` — pin each rank to its rail's HCA.
  - `NCCL_IB_TC=106`, `NCCL_IB_SL=<the SL mapped to your lossless queue>`.
  - `NCCL_IB_GID_INDEX=-1` (auto, preferred per NVIDIA for NCCL ≥2.21) or 3 (verified).
  - `NCCL_NET_GDR_LEVEL` — per NVIDIA NCCL docs, *"2 : Use GPU Direct RDMA when GPU and NIC are connected through PCI switches… The default value is 2."* Your XE9680 GPU *n*↔NIC *n* are PIX, so default 2 already enables GPUDirect RDMA; you can set `PIX`/`5` explicitly but verify the topology is detected.
  - `NCCL_IB_PCI_RELAXED_ORDERING=1` (or default auto) — large measured wins; Microsoft's test on two Standard_ND96amsr_A100_v4 VMs found *"the maximum in-place busbw with RO disabled is only 25GB/s… it can deliver 188 GB/s in-place busbw with RO enabled… more than 6X performance improvement with RO enabled"* (Azure HPC blog, "NCCL Performance Impact with PCIe Relaxed Ordering").
  - `NCCL_IB_QPS_PER_CONNECTION` (>1) and `NCCL_IB_SPLIT_DATA_ON_QPS` — useful for routing entropy on multi-tier Clos fabrics (more QPs → better ECMP spread). On a pure single-hop rail (leaf only), the benefit is small; raise it if you add a spine tier.
  - Note `NCCL_IB_ADAPTIVE_ROUTING` is disabled by default on RoCE.
- **NIXL/UCX (KV-cache):** UCX selects IPv4 RoCEv2 by default and its traffic-class auto-default is 106 — so it matches your NCCL config out of the box. Pin per-rail with `UCX_NET_DEVICES=mlx5_<n>:1`; set `UCX_IB_GID_INDEX` only if auto-selection picks wrong. NIXL (used by llm-d/Dynamo for KV transfer) rides UCX, so the same device pinning and DSCP apply.

### 5. Static IPAM `routes[]` — your approach is correct; static beats whereabouts here
Inject the per-rail route via the SR-IOV NAD's IPAM block, e.g. `ipam: { "type": "static", "addresses": [{"address":"192.168.117.1/31"}], "routes": [{"dst":"<rail-n remote prefix>","gw":"<rail-n leaf /31 peer>"}] }`. The `gw` should be the `/31` leaf-port peer — correct. For a deterministic, hand-pinned per-rail `/31` scheme, **static IPAM is preferred over whereabouts**: whereabouts exists for dynamic, cluster-wide, collision-free pool allocation, which you explicitly don't want — you want fixed, known addresses per link. Use `host-local` or `static`; static is the cleanest for fully predetermined addressing.

### 6. GPUDirect RDMA locality on the XE9680 — confirmed PIX pairing, plus the ACS/IOMMU tension
On the Dell PowerEdge XE9680 (8× HGX H100/H200), each backend NIC sits under the **same embedded PCIe Gen5 switch as its paired GPU** — a `PIX` ("at most a single PCIe bridge") relationship confirmed by real `nvidia-smi topo -m` output and by NVIDIA's AI-Enterprise doc whose test bed was an XE9680. Dell's own description is that *"each of the eight Dell PowerEdge XE9680 PCIe slots is designed with a direct connection to one of the eight GPUs through the set of embedded PCIe switches with x16 PCIe 5.0 lanes… delivering the highest throughput coupled with the lowest latency for GPUDirect and other RDMA communication."* The 8 GPU+NIC pairs split **4+4 across the two CPU sockets / NUMA nodes**: GPU0–3 + their NICs on NUMA 0, GPU4–7 + their NICs on NUMA 1. GPUDirect peer-to-peer works within a socket and does **not** operate across sockets (the cross-socket SYS path). Your GPU *n* ↔ NIC *n* rail-optimized cabling matches this perfectly — keep each rank's CPU affinity pinned to its NIC's NUMA node.

This also maps to the Dell **"2-8-9-400"** NVIDIA Enterprise Reference Architecture the XE9680 AI Factory is endorsed against. Per NVIDIA's certification docs, the name follows *CPU-GPU-NetworkAdaptor-AverageGPUNetworkBandwidth*: **2 CPUs, 8 GPUs, 9 network adapters, 400 Gb/s average GPU network bandwidth** — i.e., 8 backend RoCE/IB NICs (one per GPU at 400 Gb/s) plus a frontend adapter. Validated backend NIC options include NVIDIA ConnectX-7, BlueField-3 SuperNIC, and Broadcom BCM57608 "Thor-2."

**BIOS/platform — the key tension for your bare-metal SR-IOV + GDR case:**
- GPUDirect RDMA P2P requires **PCIe ACS disabled** on the GPU↔NIC path (ACS forces P2P transactions up to the root complex, "causing a significant performance reduction or even a hang").
- SR-IOV VFs normally require **IOMMU (VT-d) enabled** for VF isolation. Dell's fully-validated *bare-metal* XE9680 AI recipe (reproduced in Juniper's Validated Design as "recommended by Dell for their XE9680 AI/ML server") actually disables IOMMU/ACS *and* Global SR-IOV — but you need SR-IOV, so you cannot use that recipe verbatim.
- **Resolution for SR-IOV + GDR:** keep IOMMU on (required for SR-IOV), but disable ACS specifically on the PCIe switches/ports in the GPU↔NIC path, or enable NIC **ATS** (Address Translation Services) so P2P survives with ACS on (NVIDIA's documented VM-case reconciliation: "ATS must be enabled on the NIC and ACS must be configured on the PCIe switches and root ports to allow ATS in order to achieve maximum GPUDirect RDMA performance"). Verify with `lspci -vvv | grep ACSCtl` that the relevant bridges show all flags `-`. **XE9680-specific gotcha:** disabling Virtualization Technology in BIOS does *not* clear OS-visible ACS — you may need to clear ACS at the OS level each boot (`setpci` on the bridge ACS capability) via a systemd unit run before workloads.
- Set System Profile = Performance, keep PCIe relaxed ordering enabled, and ensure `nvidia-peermem` (or DMA-BUF) is loaded so GDR is actually active.

### 7. Specific failure modes of the flat `/24` and the recommended fix
- **ECMP collapse:** any namespace that holds all 8 routes to the same `192.168.0.0/24` (host netns, multi-NAD pod) will ECMP-hash flows across all 8 rails, breaking rail isolation and `NCCL_CROSS_NIC=0`, and scattering a single QP's traffic across rails with mismatched latency/PFC domains.
- **Wrong-source-NIC selection:** with one flat subnet, RoCE/RDMA-CM source-address selection can pick the wrong rail's GID for a given destination (the classic NCCL #1109 symptom), requiring fragile Linux `ip rule`/source-routing hacks to fix.
- **Overlap ambiguity:** if two remote endpoints from different rails ever land in the same `/24`, a multi-rail namespace cannot disambiguate which next-hop to use.
- **Fix:** the per-rail distinct-prefix plan in Finding 1. It eliminates all three failure modes, enforces rail isolation at L3, and costs nothing but configuration on a fabric you've said you can freely re-segment.

## Details
**Summary verdict:** Keep the routed `/31`-per-link BGP fabric, the leaf-port gateway, the static IPAM, `numVfs:1`/`netdevice`/`isRdma:true`/MTU 9000, DSCP 26 / TC 106, GID 3, and `NCCL_CROSS_NIC=0` — these are all correct, and TC 106 in particular is the canonical RoCEv2 value, not a bug. **Change two things:** (1) re-segment the remote destination into a distinct prefix per rail index instead of one shared flat `/24`; (2) make sure the platform plumbing for maximum performance is in place — RDMA `exclusive` mode, verified same-PCIe-switch GDR with ACS handled correctly, relaxed ordering on, end-to-end MTU 9000 + DSCP/PFC/ECN consistency, and ECN/DCQCN as primary congestion control.

The performance ceiling on this design is set far more by the platform plumbing (GDR P2P actually engaging, relaxed ordering, lossless config consistency, NUMA pinning) than by whether the remote route is one /24 or eight prefixes. The flat /24 is a latent correctness/robustness risk rather than a throughput bug in the strict one-rail-per-pod case — but since you're optimizing for zero risk of congestion/cross-rail leakage and you can re-segment freely, do it.

## Recommendations
**Stage 1 — Re-segment routing (do first, low risk):**
1. Allocate one remote prefix per rail index on both OS and switch sides; rewrite each rail's NAD to carry only `dst <rail-n remote prefix> via <rail-n leaf /31 peer>`.
2. Verify in a pod: `ip route` shows exactly one route; `show_gids` confirms index 3 = RoCEv2 IPv4; `ib_write_bw`/`ib_send_bw` still pass per rail.
3. Benchmark `all_reduce_perf` and `alltoall` before/after — busbw should be unchanged or better, with no cross-rail traffic in `NCCL_DEBUG=INFO` logs.

**Stage 2 — Lock in platform performance:**
4. Set `rdmaMode: exclusive` in `SriovNetworkPoolConfig`; confirm each pod gets an isolated RDMA device.
5. Confirm GDR P2P actually engages: check `nvidia-smi topo -m` shows PIX for each GPU↔NIC pair; check `lspci -vvv | grep ACSCtl` shows ACS off on the GPU↔NIC bridges; confirm `nvidia-peermem`/DMA-BUF loaded. Resolve the SR-IOV-needs-IOMMU vs GDR-wants-ACS-off tension via ATS or path-specific ACS disable.
6. Verify MTU 9000 end-to-end (pod, VF, every leaf port at ≥9216) and DSCP-26→lossless-queue + DSCP-48(CNP)→strict-priority mapping is identical on NICs and all switches.
7. Set NCCL env: `NCCL_CROSS_NIC=0`, `NCCL_IB_HCA` per rail, `NCCL_IB_TC=106`, `NCCL_IB_GID_INDEX=-1` (auto) or 3 (verified), `NCCL_IB_PCI_RELAXED_ORDERING=1`, `NCCL_NET_GDR_LEVEL=PIX`.

**Stage 3 — Tune congestion control & scale-out:**
8. Run ECN/DCQCN as primary; tune WRED/ECN thresholds below PFC threshold; monitor `ethtool -S | grep ecn` and switch PFC/ECN counters. Zero PFC pauses under load is the goal.
9. If/when you add a spine tier, raise `NCCL_IB_QPS_PER_CONNECTION` and enable `NCCL_IB_SPLIT_DATA_ON_QPS` for ECMP entropy.

**Thresholds that change the recommendation:**
- If you ever need multi-rail-in-one-pod (PXN aggregation, multi-NIC NIXL), the per-rail-prefix re-segmentation becomes mandatory, not optional.
- If `NCCL_DEBUG=INFO` shows cross-rail communication or GID-selection warnings, hard-pin GID index and source routing.
- If you see PFC pause storms or non-zero tail latency, ECN thresholds are mistuned or a hop's MTU/DSCP map is inconsistent.

## Caveats
- GID index 3 is correct for mlx5 with a single IPv4, but is firmware/IP-count dependent and can shift on link flap — always verify inside the pod and prefer NCCL auto-selection where possible.
- The "Dell recommends disable IOMMU/ACS" bare-metal BIOS recipe disables SR-IOV entirely; because you require SR-IOV, you must reconcile the IOMMU-on/ACS-handling tension yourself (ATS or path-specific ACS disable) — validate empirically with a GDR bandwidth test, since exact behavior is silicon/firmware specific.
- PFC across L3 routed hops works but is more fragile than on L2; rely on ECN/DCQCN and treat PFC as a safety net.
- Benchmarks cited (e.g., the ~6× relaxed-ordering gain) come from A100/Azure VM environments; your bare-metal H100/H200 numbers will differ — measure on your hardware.
- This analysis assumes NVIDIA HGX (ConnectX-7/BlueField-3 or mlx5-class) NICs; the AMD MI300X XE9680 variant does *not* use a hard 1:1 GPU:NIC PIX mapping and would change the rail-pinning advice.
