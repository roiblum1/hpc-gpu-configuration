## 1. Topology: why 4 groups Ã— 2 nodes, not one big group or eight small ones

MiniMax M2.7 is a 230B-total / 10B-active MoE (256 experts, top-8) that fits comfortably in FP8 on a single 8Ã—H100 node. That single fact drives everything: nothing *forces* multi-node parallelism, so any node-spanning design has to earn its keep. Three layouts were on the table.

The first is one giant group (`size: 8`, one replica) â€” TP or EP spanning all 64 GPUs. This is rejected outright: a single node failure takes down 100% of serving, rolling updates mean total outage, and TP=64 over IB puts inter-node latency into every single decode step, which is poison for speculative decoding. It fails the survivability requirement completely.

The second is eight single-node replicas (`size: 1`, `replicas: 8`) â€” pure data parallelism, each node an island, TP=8 over NVLink. This is the maximum-survivability, minimum-complexity design: lose a node, lose 12.5% of capacity, and DFlash never touches the fabric. Its weakness is that it leaves the IB/RDMA requirement unused (IB carries nothing) and it is not the throughput-optimal shape for a 256-expert MoE at high concurrency, because every node must hold all 256 experts, which eats HBM that could otherwise be KV cache.

The chosen design is the middle path: `size: 2`, `replicas: 4`. Inside each node, TP=8 over NVLink exactly as before. Across the two nodes of a group, vLLM runs in data-parallel mode (`--data-parallel-size 2`) with `--enable-expert-parallel`, which shards the MoE expert weights EP=16-wide across all 16 GPUs of the group. Attention and KV cache stay node-local (DP-replicated); only the MoE dispatch/combine all-to-all crosses the IB fabric, via DeepEP in its low-latency mode. This is the DeepSeek-style "wide-EP" serving shape, scaled down to a 2-node group.

Why this earns its complexity: halving the expert weights per GPU frees roughly 14â€“15 GB of HBM per GPU (see Â§2), and every freed gigabyte becomes KV cache, which directly becomes larger concurrent batch capacity â€” exactly the resource in short supply during the daytime peak. The all-to-all is latency-tolerant compared to TP allreduce (it happens once per MoE layer on a small activation payload for 10B-active tokens), and DeepEP over 400 Gb/s rails handles it well. Meanwhile the latency-critical parts of DFlash â€” draft forward and verification attention â€” remain on NVLink within each DP rank.

The honest caveat: speculative decoding combined with multi-node DP+EP is a newer code path in vLLM than either feature alone. Validate it on the exact build before committing. If the pinned vLLM version chokes on the combination, the fallback is a two-line change to the same deployment â€” `replicas: 8`, `size: 1` (in the chart: `worker.replicas: 8`, `worker.nodeCount: 1`), delete the `--data-parallel-*` and `--enable-expert-parallel` flags â€” and you land safely on the second layout with zero re-architecture. That is deliberately why the design uses gang machinery (LWS originally, Grove in the chart) even though single-node groups don't strictly need it.

## 2. Memory and capacity math

FP8 weights are roughly 230 GB. In the single-node layout, each GPU carries ~29 GB of weights, leaving ~45 GB for KV cache at 90% memory utilization on an 80 GB H100. In the wide-EP layout, the ~200 GB of expert weights are sharded 16 ways instead of 8, so per-GPU weight footprint drops to roughly 15â€“17 GB (dense layers + attention still TP-8 replicated across the DP pair, experts halved). That yields ~57â€“60 GB of KV per GPU â€” a ~30% larger KV budget, which at FP8 KV (`--kv-cache-dtype fp8_e4m3`) translates into proportionally more concurrent sequences at the 196K max context, or the same concurrency with far more prefix-cache headroom. `--max-num-seqs 128` is a starting point sized to that budget; tune it against observed `gpu_cache_usage_perc` in the existing Grafana stack.

Note the asymmetry with the GLM-5.1 deployment: GLM at 754B *needs* multi-node or aggressive quantization; M2.7 doesn't, so here wide-EP is a throughput optimization, not a requirement â€” which is why the escape hatch back to single-node must stay cheap.

## 3. RDMA data path

Three separate transports exist in this design and it matters not to conflate them. Intra-node TP runs on NVLink/NVSwitch through NCCL and never touches a NIC. Inter-node EP all-to-all runs on DeepEP (`VLLM_ALL2ALL_BACKEND=deepep_low_latency`), which uses NVSHMEM/IBGDA over the ConnectX-7 rails with GPUDirect RDMA â€” this requires `nvidia-peermem` (or DMA-BUF on newer stacks) and GDRCopy loaded on the hosts, which the GPU Operator setup should already provide. Control-plane bootstrap (torch distributed init, DP RPC) runs over the pod network, which is why `NCCL_SOCKET_IFNAME`/`GLOO_SOCKET_IFNAME` pin to `eth0` while `NCCL_IB_HCA` enumerates only the eight compute-rail HCAs â€” excluding the storage/north-south CX-7s is important, otherwise NCCL will happily register QPs on the storage fabric.

The pod attaches one VF per rail through the existing Multus NADs, preserving the GPUâ†”NIC PCIe-switch pairing already engineered for the RoCEv2 work â€” same rail discipline, IB flavor. `IPC_LOCK` capability is required for RDMA memory registration (in this repo it is covered fleet-wide by node-foundation's CRI-O `memlock=-1` MachineConfig); nothing here needs `privileged`.

The `leaderworkerset.sigs.k8s.io/exclusive-topology` annotation is the fabric-aware scheduling piece: it forces both pods of a group onto nodes sharing the same topology label (leaf/block), so the EP all-to-all never pays a spine hop. (In the chart translation this job moves to KAI's network-topology-aware placement.) Label the nodes per leaf and point the placement at that key.

`NCCL_IB_TIMEOUT=22` and `NCCL_IB_RETRY_CNT=13` are deliberately generous: with EP spanning nodes, a transient link flap that would otherwise abort the communicator (and thus the whole group) instead stalls and recovers. This trades tail latency during a flap for group survival â€” the right trade for the survivability requirement.

## 4. NUMA alignment

A DGX H100 has two Xeon 8480C sockets with GPUs 0â€“3 and their four paired CX-7 rails on socket 0, GPUs 4â€“7 and theirs on socket 1. Since each pod consumes the entire node (8 GPUs), the classic Kubernetes `single-numa-node` topology policy is mathematically impossible â€” the pod's devices span both sockets by construction. What NUMA alignment *means* for a whole-node pod is different: it means the per-GPU worker processes, their pinned host memory, and their RDMA buffers each stay on the socket of their GPU, and no worker's memory lands on the remote socket.

The deployment does its part by making the pod Guaranteed QoS with integer CPUs (`cpu: "96"`, requests == limits), which activates static CPU-manager pinning. The node does its part through kubelet policy â€” on OCP 4.20 the clean way is a PerformanceProfile (in this repo: the node-foundation chart, which keeps `topologyPolicy: best-effort` with reserved cores on both sockets for the same full-node reason). `restricted` would also admit the inevitable dual-socket pod while forcing co-location where possible; the repo's chosen policy is `best-effort` with correctness delegated to the runtime. Inside the container, vLLM/NCCL read the PCIe topology and place each TP worker adjacent to its GPU; DeepEP registers its buffers against the rail HCA on the same PCIe switch as the GPU, so GPUDirect traffic never crosses the socket interconnect. The remaining lever, if `numastat` shows cross-socket memory, is wrapping worker spawn with `numactl --membind` per rank â€” but with static CPU manager and Memory Manager `Static`, that is rarely needed. This is a care-about rather than a must precisely because the marginal win over a well-pinned default is a few percent; the config above captures most of it for free.

## 5. Surviving node loss

The blast radius arithmetic: with 4 groups of 2, an unplanned node death (power cut) kills exactly one group. `RecreateGroupOnPodRestart` (Grove's atomic gang recreate in the chart) tears down the surviving peer immediately rather than letting it spin on a dead NCCL communicator â€” a half-alive EP group can never heal, so fast atomic recreation beats optimistic waiting. The router drops the dead group within seconds and clients fail over to the three healthy groups at 75% capacity. Because there are exactly 8 nodes with no spares, the recreated group stays Pending until the node returns, then rejoins automatically â€” no operator action needed beyond fixing the node.

Compare the fallback single-node layout: the same event costs 12.5% instead of 25%. That is the real price of wide-EP, and it is why the choice between the two layouts should ultimately be settled by measurement: if wide-EP buys more than ~15% peak throughput (plausible given the KV math in Â§2), it wins even accounting for the worse failure quantum; if it buys less, take the 8Ã—1 layout and the better blast radius.

For *planned* disruption (maintenance), the PodDisruptionBudget guarantees a drain can never take a second group down while one is already out, and the rolling-update policy (`maxUnavailable: 1`, `maxSurge: 0`) applies the same discipline to image/config rollouts â€” upgrades run at 75% capacity, one group at a time, which slots naturally into the night window when speculation is carrying the load anyway.

Two things this design consciously does not do: it does not try to make a group survive the loss of one of its members (elastic EP resize mid-flight is not something vLLM supports), and it does not add a spare node's worth of surge capacity, because there is no ninth node. If a fifth nine matters later, the cheapest insurance is running one group's worth of traffic through the 4-node environment as an overflow target behind the same gateway.

## 6. DFlash specifics, the draft model, and the diurnal pattern

The `speculative-config` block requests `method: dflash` with `num_speculative_tokens: 8` â€” DFlash's economics favor high K because all K draft tokens cost one forward pass, unlike EAGLE-style autoregressive drafting. `draft_tensor_parallel_size: 1` keeps the small draft on a single GPU per rank.

**Draft availability (updated for this branch):** a public draft exists â€” [z-lab/MiniMax-M2.7-DFlash](https://huggingface.co/z-lab/MiniMax-M2.7-DFlash) â€” and is mirrored + staged by the model-staging chart, so **DFlash is the day-one default in this environment** (it is used *only* in the 8-node environment; the 4-node environment runs MTP). The zero-cost fallback remains available and unchanged: M2.7 ships native MTP heads in the checkpoint, so `{"method":"mtp","num_speculative_tokens":3}` gives co-trained speculative decoding with zero extra VRAM and no draft artifact to manage. If DFlash underperforms on the measured traffic, flip `speculative.method: mtp` and compare acceptance length.

`disable_by_batch_size: 32` is what makes the day/night pattern self-managing: the engine evaluates batch size per step, so at night (small batches, bandwidth-bound decode) speculation is active and delivers the 2â€“3Ã— latency win, while at daytime peak (large batches, compute-bound) it silently disables and the engine reverts to raw batched throughput â€” which is precisely when wide-EP's extra KV headroom is doing the work instead. The two mechanisms are complementary across the diurnal cycle rather than redundant. Verify the exact key name against the pinned vLLM version; this knob has moved between releases.

## 7. Operational checklist for the air-gapped environment

Mirror an image that contains vLLM with DeepEP and DeepGEMM compiled in (the stock `vllm/vllm-openai` image plus a DeepEP layer is the usual recipe â€” same pattern as the Dynamo mirror script). Confirm on the hosts: `nvidia-peermem` loaded, GDRCopy present, and `ibstat` shows all eight compute rails `Active` (a `Polling` rail has bitten before â€” same check applies). Pre-stage weights and the draft model on local NVMe on all eight nodes (the model-staging chart); a 230 GB checkpoint over the fabric on every group restart is avoidable pain. Expect first startup in the tens of minutes (weight load + CUDA graph capture) â€” the startupProbe budget reflects that; don't tighten it. Validate the full stack in order: single node TP=8 no speculation â†’ single node + MTP â†’ 2-node group DP+EP no speculation â†’ full config. When something breaks, that ladder tells you which layer did. Finally, wire `vllm:spec_decode_num_accepted_tokens_total` and the DeepEP dispatch latency into the RDMA Grafana dashboard â€” acceptance length and all-to-all p99 are the two numbers that will tell you whether this topology is earning its keep.
